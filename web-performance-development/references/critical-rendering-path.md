# The critical rendering path

Everything between the first byte of HTML and pixels on screen. The
discipline: know what blocks rendering, keep that set minimal, and start the
critical fetches as early as physics allows.

## The pipeline

HTML parse → DOM. CSS fetch/parse → CSSOM. DOM + CSSOM → render tree →
layout → paint → composite. Two facts drive everything else:

- **CSS is render-blocking**: the browser won't paint until the CSSOM is
  complete — every stylesheet in `<head>` delays first paint.
- **Synchronous `<script>` is parser-blocking**: the parser stops, fetches,
  executes, then resumes — and it also waits for any pending CSS (script
  might read styles).

The **preload scanner** mitigates both: a lookahead parser that starts
fetches for resources in the raw HTML even while the main parser is blocked.
Resources it can't see — CSS `background-image`, anything constructed by JS —
start late; that's most "LCP load delay" problems.

## TTFB — the floor under everything

Nothing renders before the HTML arrives. Budget ~800 ms at p75; spend it
deliberately: DNS + TLS (fewer origins, `preconnect`), redirects (each one is
a full round trip — kill chains), server think-time (cache pages or
fragments; slow queries → `sql-development`), and CDN cache hits for
anything cacheable (`caching-and-cdn.md`). `Server-Timing` headers expose the
origin-side split to RUM.

**103 Early Hints** lets the origin flush `Link: rel=preload/preconnect`
hints while still composing the page — meaningful when server think-time is
irreducible. CDN + browser support is partial (Chromium and Safari at
different completeness; July 2026 — re-verify your chain end to end).

## CSS strategy

- Ship less: dead-CSS elimination, per-route splitting where the toolchain
  allows.
- **Critical CSS**: inline the above-the-fold rules (~≤ 14 KB compressed) in
  `<head>` and load the rest without blocking:

```html
<style>/* critical above-the-fold rules, generated by tooling */</style>
<link rel="preload" href="/css/site.css" as="style"
      onload="this.onload=null;this.rel='stylesheet'">
<noscript><link rel="stylesheet" href="/css/site.css"></noscript>
```

  Worth the tooling on slow-network audiences; skip if your full CSS is
  already small.
- Split by media: `<link media="print">` and conditional viewports don't
  block rendering when they don't match.
- Never `@import` in CSS — it serialises stylesheet discovery.
- `font-display` and font preloading interact with first paint —
  `images-fonts-media.md`.

## JavaScript strategy

| Loading | Behaviour | Use |
|---|---|---|
| `<script>` (bare, in body/head) | Blocks parsing at that point | Only the rare script that must run before content below it |
| `defer` | Fetch in parallel; execute in order after parse, before DOMContentLoaded | Default for classic scripts |
| `async` | Fetch in parallel; execute whenever ready (unordered, can interrupt parse) | Independent scripts: analytics, ads |
| `type="module"` | Deferred by default; use `modulepreload` for its dependency tree | Modern default |

Nothing synchronous in `<head>` that isn't genuinely critical; third-party
tags are `async`/deferred or façaded (`loading-and-code-splitting.md`).
`document.write` is banned outright.

## Transfer

- **Brotli** for text assets (falls back to gzip); precompress statics at
  build time rather than per-request. zstd encoding exists in some
  browser/CDN pairs (July 2026 — decision-level, verify support before
  relying on it).
- HTTP/2+ multiplexing means bundling-for-connection-reuse is dead as a
  motive — bundle for compression efficiency and request-count sanity, split
  for caching and code-splitting (`loading-and-code-splitting.md`).
- Minify HTML/CSS/JS in the build; strip source maps from production
  payloads (serve them out-of-band).

## Order of operations in `<head>`

A pragmatic ordering that avoids self-inflicted blocking: meta/charset/
viewport → title → inline critical CSS → preconnects for critical origins →
preloads for critical late-discovered assets → blocking CSS links → deferred/
module scripts → everything else. Audit with a DevTools performance trace:
the request waterfall makes ordering mistakes visible immediately —
verify rather than assume.
